## 1. Resumo executivo

**semidx** é busca semântica **self-hosted** para código e documentos: chunk → embed → store → resultados ranqueados em `arquivo:linha`. O binário principal `semidx` oferece CLI, servidor HTTP + admin SPA e MCP; o repositório também produz o binário complementar `chatrag`.

O projeto saiu claramente do estágio “PoC de homelab”. A base de engenharia (backends de storage, chain de embedding com privacy, hybrid search, jobs e CI) é sólida. O próximo salto de valor não é “mais features soltas”, e sim:

1. **Fechar os gates declarados** — cobertura ≥90% e testes do frontend
2. **Eliminar drift operacional** entre código, requirements, AGENTS e harness MCP
3. **Paridade e foco** entre CLI · API · Admin · MCP
4. **Validar escala com corpus real**, SLOs e limites por projeto

| Dimensão | Nota | Comentário |
|----------|------|------------|
| Visão de produto | **A** | Nicho claro: self-host + privacy + agents |
| Arquitetura | **A-** | Split store, privacy routing, jobs, 3 backends |
| Search quality | **B+** | Hybrid RRF, Graph-RAG, rerank lexical; AST/eval ainda parciais |
| Escala | **B** | Homelab/mid-size ok; path 1M+ chunks desenhado, pouco hardenizado |
| Superfícies (CLI/API/UI/MCP) | **B-** | CLI à frente; UI e MCP cobrem o core, não a borda |
| Segurança | **B+** | Core API maduro; caps, chat entrypoints e quotas merecem revisão |
| DX / CI | **B+** | Gates fortes, mas cobertura está abaixo da meta e o frontend não tem testes |
| Documentação | **B** | Boa cobertura, porém requirements/research/harness têm status defasados |

**Veredito:** utilizável hoje para dev solo, homelab e time pequeno self-hosted, com integração de agentes acima da média. Ainda não deve ser tratado como operação crítica ou comprovadamente pronta para grande escala: faltam fechar o gate de cobertura, testar a SPA e validar os limites com massa real. Não precisa reescrever a arquitetura.

## 2. O que o produto faz

### 2.1 Problema que resolve

Ferramentas de “semantic code search” costumam ser:

- **SaaS** (código e chaves saem da rede), ou
- **Local-only** (sem índice compartilhado para o time / agentes).

semidx ocupa o **meio self-hosted**:

- Índice central no servidor **ou** SQLite no laptop
- Chaves de embedding ficam no host que embute (servidor ou máquina local)
- Arquivos sensíveis não vão para cloud (privacy routing)
- Funciona **sem modelo** (`--keyword`)
- MCP + skills para agentes

**Não é:** substituto de `rg`/grep, IDE completa, SaaS multi-tenant, nem análise de PR estilo Sonar em todas as branches.

### 2.2 Capacidades principais

| Área | Capacidade |
|------|------------|
| Index | Git (identity estável), pastas `--docs`, JAR/WAR/AAR, hash-diff incremental |
| Extract | Código/texto, Markdown/HTML, PDF, OOXML, ODF, EPUB, Jupyter, RTF, `.class` (+ decompiler opcional) |
| Embed | Chain Gemini → Groq → OpenRouter → Ollama Cloud → Ollama local; privacy force-local / text-only |
| Store | Postgres/pgvector (HNSW), SQLite local, ou proxy remoto (`login`) |
| Search | Vetorial, keyword, hybrid RRF (default), fallback explícito, worktree filter, Graph-RAG, reranker lexical opcional |
| Server | API `/api/v1`, jobs de index, git-sync, push (raw ou embed-local), admin SPA |
| Agents | MCP stdio, `mcp install` (~12 clientes), skills embutidas |
| Extras CLI | callers, explain, dead-code, diff, sbom, secrets, alerts, insights, bench, init, upgrade, chatrag |

### 2.3 Formas de uso

```
Standalone                          Client / Server
─────────────────                   ────────────────────────────────
semidx --local index/search         semidx CLI / MCP ──HTTP──▶ serve
        │                                      │
        ▼                                      ▼
   SQLite index.db                    Postgres/pgvector + /admin SPA
```

Precedência de backend (runtime): **remote > Postgres (configurado) > SQLite > Postgres (default)**.

## 3. Arquitetura (visão atual)

### 3.1 Layout

```
cmd/semidx/          CLI cobra + serve + mcp
cmd/chatrag/         Chat RAG standalone / serve
pkg/client/          SDK HTTP público
internal/
  config/            SEMIDX_* (+ .env + user env)
  clientconfig/      server_url, token, default project
  embed/             Embedder + Chain + circuit breaker
  privacy/           deny-list / route-local
  extract/           registry pluggable de formatos
  chunker/           chunks com linha (+ AST parcial)
  indexing/          walk, hash-diff, worktree, watcher
  search/            hybrid, routing, graph, rerank, format
  store/             PgStore + migrations goose
  localstore/        SQLite WAL, top-K heap, FTS5
  server/            API + jobs + auth scopes
  webadmin/ webui/   admin legado + SPA embutida
  mcpserver/         tools MCP sobre Backend remote|local
  rag/ webchat/      pipeline e UI de chat
deploy/              compose, homelab, agentics-test
docs/                architecture, requirements, research, ADRs
web/                 fonte Vite/React da SPA
```

### 3.2 Pipeline de indexação e busca

```
fonte (git / pasta / JAR / push)
        │
        ▼
  extract  →  chunk  →  privacy.IsSensitive?
        │                    │
        │              sim ──┼── não
        │                    │
        ▼                    ▼
   local / text-only    chain de embed (cloud ok)
        │                    │
        └────────┬───────────┘
                 ▼
          IndexStore (Pg | SQLite)
                 │
                 ▼
   search: hybrid (vector ‖ keyword → RRF)
           → opcional graph expand / rerank
           → fallback keyword se embed falhar
```

### 3.3 Decisões que valem ouro

| Decisão | Por quê importa |
|---------|-----------------|
| `IndexStore` vs `Store` | Standalone não implementa tokens/users/jobs |
| Tabelas `chunks_<dims>` | Modelos de dimensões diferentes coexistem |
| HNSW + halfvec em dims altas | ANN acima do limite 2000 do tipo `vector` |
| Content-addressed por hash | Re-embed só o que mudou |
| Worktree manifest | Busca o snapshot do worktree, não “qualquer path do repo” |
| Privacy no path de embed | Credenciais e `.env` não vão para API cloud |
| Keyword-only como modo de produto | Onboarding sem GPU/API key |

### 3.4 Escala do repositório (ordem de grandeza)

| Métrica | Aprox. |
|---------|--------|
| LOC Go (excl. `web/node_modules`) | ~60k |
| Arquivos `.go` | ~300+ |
| Arquivos `*_test.go` | ~160+ |
| Pacotes `internal/` | ~35 |
| Comandos CLI | dezenas (core + análise + productivity) |

## 5. Pontos fortes

1. **Posicionamento** — self-host + privacy + agents, sem prometer “ser o Zoekt”.
2. **Arquitetura de backends** — laptop (SQLite) e time (Postgres) honestos.
3. **Search em evolução** — hybrid default, graph caps, rerank seam, `semidx bench`.
4. **Extractors** — cobertura de documentos bem acima de indexers “só código”.
5. **Agentes** — MCP + install multi-cliente + skills embutidas.
6. **Qualidade de engenharia** — race tests, lint, gosec, govulncheck, mutation testing e PBT já fazem parte da fábrica.
7. **Documentação de produto** — requirements com IDs estáveis e matriz de paridade, embora precisem de reconciliação periódica.

## 7. Roadmap recomendado

Alinhado a `requirements.md` §§6 (fases A–E) e ao research de escala, reordenado por **alavancagem prática**.

### Fase 1 — Fechar qualidade e fonte da verdade

1. Corrigir AGENTS/comentários/harness do `mcp install --client codex --apply`
2. Reconciliar `requirements.md` e o research com o código entregue
3. Elevar cobertura Sonar de ~84,8% para ≥90%, priorizando store/webadmin/indexing/server
4. Adicionar Vitest + Testing Library e um E2E Playwright do fluxo login → ingest → job → search
5. Corrigir/workaround do trigger CI em `main`

**Critério de saída:** todos os gates declarados são executáveis e verdes; documentação e harness descrevem o comportamento real.

### Fase 2 — Fechar o loop diário (produto)

1. Polir Admin: file detail, ações de tabela, empty states (`REQ-UI-06/08`)
2. Política explícita: o que fica CLI-only (dead-code/SBOM/alerts) e o que sobe à UI (`REQ-UI-10`)
3. Quotas persistidas por projeto, aplicadas em repo jobs, push raw/pre-embedded e browser ingest
4. Revisar auth e isolamento em chat/RAG e rotas secundárias

**Critério de saída:** um usuário admin consegue indexar, acompanhar job, buscar e inspecionar arquivo/grafo sem CLI, com limites previsíveis.

### Fase 3 — Search quality e escala medida

1. Expandir AST chunking (Go, TypeScript/JavaScript, Python primeiro)
2. `semidx bench` em nightly ou subset no CI, com baseline versionada
3. Medir p50/p95, recall e custo em 50k, 500k e 1M chunks
4. Avaliar halfvec default e dedup cross-project com dados medidos
5. Aviso automático “prefira Postgres” acima de limiar validado de chunks
6. Opcional: cross-encoder local no seam de rerank

**Critério de saída:** regressões de recall/latência são detectáveis e as recomendações SQLite/Postgres têm evidência reproduzível.

### Fase 4 — Fábrica e escala avançada

1. Decompor as implementações grandes de store/indexing por domínio
2. OpenAPI como fonte única ou contract tests automatizados
3. Dedup global de conteúdo
4. Só então: quantização binária, hot/cold, PQ
